# 20.2 — Java 12 to 16 Features

> **Versions covered:** Java 12 (2019) · Java 13 (2019) · Java 14 (2020) · Java 15 (2020) · Java 16 (2021)
>
> This era delivered the most significant language-level changes since Java 8. Switch expressions replaced a clunky statement. Text blocks ended string-literal formatting nightmares. Records eliminated the boilerplate data class. Pattern matching for `instanceof` made casts expressive and safe. Sealed classes gave type hierarchies a formal contract. All of these features graduated from preview to stable across these five releases — Java 17 LTS shipped all of them fully stable, so understanding their evolution helps you reason about them correctly.

---

## ☕ Java 12 & 13 — Switch Expressions

### 20.2.1 Switch Expressions

The classic `switch` statement was one of Java's most error-prone constructs. Its fall-through behaviour required careful discipline, and because it was a *statement* rather than an *expression*, you couldn't use it on the right side of an assignment without duplicating logic. Java 12 introduced switch *expressions* as a preview, and they became stable in Java 14.

To understand why this matters, look at the classic problem:

```java
// ❌ Classic switch statement — three problems:
// 1. Fall-through: forgetting a break causes silent bugs
// 2. It's a statement, not an expression — can't be used inline
// 3. Each case needs a separate assignment to a pre-declared variable

String day = "WEDNESDAY";
int numLetters;

switch (day) {
    case "MONDAY":
    case "FRIDAY":
    case "SUNDAY":
        numLetters = 6;
        break;                    // Easy to forget this break
    case "TUESDAY":
        numLetters = 7;
        break;
    case "THURSDAY":
    case "SATURDAY":
        numLetters = 8;
        break;
    case "WEDNESDAY":
        numLetters = 9;
        break;
    default:
        throw new IllegalArgumentException("Not a day: " + day);
}
```

The new **arrow-syntax switch expression** solves all three problems simultaneously:

```java
// ✅ Java 14+ switch expression — arrow syntax
// Each arm is an expression that produces a value. No fall-through. No break.
// The whole switch is itself an expression, so you can assign it directly.

String day = "WEDNESDAY";

int numLetters = switch (day) {
    case "MONDAY", "FRIDAY", "SUNDAY"     -> 6;  // Multiple labels in one arm
    case "TUESDAY"                         -> 7;
    case "THURSDAY", "SATURDAY"           -> 8;
    case "WEDNESDAY"                       -> 9;
    default -> throw new IllegalArgumentException("Not a day: " + day);
    // Note: if all enum values are covered, default is not required
};
```

When a case arm needs multiple statements (not just a single expression), you use a **block** with the `yield` keyword to produce the value:

```java
// yield is used when the case arm is a block, not just a single expression
double discount = switch (customerTier) {
    case PLATINUM -> 0.20;
    case GOLD     -> 0.15;
    case SILVER   -> 0.10;
    case STANDARD -> {
        // Multiple statements are fine in a block arm
        double base = 0.05;
        double bonus = isLoyalCustomer(customerId) ? 0.02 : 0.0;
        yield base + bonus;  // yield replaces return — it produces the block's value
    }
};
```

Switch expressions also work on `String`, `int`, `long`, `char`, `byte`, `short`, their boxed types, and **enums**. With Java 21's pattern matching in switch, they also work on any object type — that's covered in section 20.3.

**Important: switch expressions must be exhaustive.** The compiler verifies that every possible input value has a case. If you switch on an `enum`, you must either cover every constant or provide a `default`. This compile-time exhaustiveness check is a major safety improvement over the old statement form.

```java
// The compiler ensures this is exhaustive when switching on an enum
// Remove any case and you get a compile error — not a runtime surprise
Season description = switch (season) {
    case SPRING -> Season.WARM;
    case SUMMER -> Season.HOT;
    case AUTUMN -> Season.COOL;
    case WINTER -> Season.COLD;
    // No default needed — all enum constants covered
};
```

> **Best Practice:** Prefer switch expressions over switch statements for almost all new code. The arrow syntax eliminates fall-through bugs, forces exhaustiveness, and makes the code more functional and composable. Reserve the old colon syntax only when you genuinely need fall-through behaviour (rare) or when maintaining legacy code.

### 20.2.2 `String.indent()` and `String.transform()` (Java 12)

Two small but useful String helpers arrived in Java 12.

`indent(int n)` adds or removes leading whitespace on each line of a multi-line string and normalises line endings. Positive `n` adds spaces, negative `n` removes them (up to available whitespace).

```java
String raw = "Hello\nWorld\nJava";

// Add 4 spaces of indentation to each line
String indented = raw.indent(4);
// Result:
//     Hello
//     World
//     Java
// (Note: also adds a trailing newline if there wasn't one)

// Remove indentation (negative n)
String dedented = "    Hello\n    World".indent(-2);
// Result:
//   Hello
//   World

// Practical use: embedding code in output
System.out.println("Configuration:\n" + config.toString().indent(2));
```

`transform(Function<String, R>)` applies a function to the String, producing any type. It enables a more fluent, pipeline-style processing without intermediate variables:

```java
// Before transform: intermediate variables break the flow
String input = "  hello world  ";
String trimmed = input.strip();
String[] words = trimmed.split(" ");
List<String> wordList = Arrays.asList(words);

// With transform: a single pipeline expression
List<String> wordList = "  hello world  "
    .transform(String::strip)
    .transform(s -> s.split(" "))
    .transform(Arrays::asList);
    // Note: each transform() can return a different type
```

## ☕ Java 13 & 14 — Text Blocks and Records Preview

### 20.2.3 Text Blocks (Preview Java 13–14, Stable Java 15)

Text blocks solve one of Java's most painful daily problems: embedding multi-line strings — JSON, SQL, HTML, XML, YAML — in source code. Before text blocks, you were forced into either a chain of concatenated string literals (with escape sequences for every quote) or reading from a file.

A text block begins with a `"""` opening delimiter followed immediately by a newline, and ends with a closing `"""`. The content between them is the string — newlines, quotes, and all — without escape sequences.

```java
// ❌ Before text blocks: escaping hell for embedded JSON
String json = "{\n" +
              "    \"name\": \"Alice\",\n" +
              "    \"age\": 30,\n" +
              "    \"roles\": [\"admin\", \"user\"]\n" +
              "}";

// ✅ Java 15+ text block: the string looks exactly like what it represents
String json = """
        {
            "name": "Alice",
            "age": 30,
            "roles": ["admin", "user"]
        }
        """;

// SQL is dramatically cleaner
String sql = """
        SELECT u.id, u.name, u.email, o.total
        FROM users u
        JOIN orders o ON u.id = o.user_id
        WHERE u.region = :region
          AND o.status = 'ACTIVE'
        ORDER BY o.created_at DESC
        LIMIT 100
        """;

// HTML template
String html = """
        <!DOCTYPE html>
        <html>
          <head><title>%s</title></head>
          <body>
            <h1>%s</h1>
            <p>%s</p>
          </body>
        </html>
        """.formatted(title, heading, body);
```

**Understanding indentation handling** is crucial to using text blocks correctly. The compiler automatically strips a common leading whitespace prefix from all lines — determined by the position of the closing `"""`. This means the indentation in your source code doesn't become part of the string value.

```java
// The position of the closing """ controls how much leading whitespace is stripped.
// Here, closing """ is at column 8, so 8 spaces are stripped from every line.
String block1 = """
        Line 1
        Line 2
        """;
// Value: "Line 1\nLine 2\n"  (8 leading spaces removed from each line)

// Moving the closing """ to column 0 changes nothing here (already fully stripped).
// But if you want to PRESERVE leading spaces in the content, close at a column
// that is LESS than the content's indentation.
String block2 = """
        Line 1
        Line 2
""";
// Value: "        Line 1\n        Line 2\n"  (all 8 spaces preserved — closing """ is at column 0)

// Note: The trailing newline from the last content line is always included
// because the closing """ is on its own line.
// If you DON'T want a trailing newline, put the closing """ on the same line as the last content:
String noTrailingNewline = """
        Hello""";
// Value: "        Hello"  (no trailing newline, leading spaces not stripped)
```

**Escape sequences in text blocks** still work when you need them:

```java
// \n, \t, \\ etc. still work
// But you also get two new text-block-specific escape sequences:

// \s — explicitly adds a single space, prevents whitespace trimming
// Useful to preserve trailing spaces on a line (which would otherwise be stripped)
String withTrailingSpace = """
        line one   \s
        line two
        """;

// \<line-ending> — line continuation — suppresses the newline at the end of the line
// Useful for very long lines you want to break visually but keep as one logical line
String longLine = """
        This is a very long string that spans \
        multiple source code lines but \
        is actually one continuous string.
        """;
// Value: "This is a very long string that spans multiple source code lines but is actually one continuous string.\n"
```

### 20.2.4 Records (Preview Java 14–15, Stable Java 16)

Records are Java's answer to the endless boilerplate of data carrier classes — POJOs that exist purely to hold and transport data. Before records, a simple data class required: field declarations, a constructor, getters, `equals()`, `hashCode()`, and `toString()` — 50+ lines for what is conceptually a 1-line idea.

```java
// ❌ Before records: a simple immutable Point class requires ~40 lines
public final class Point {
    private final int x;
    private final int y;

    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public int x()      { return x; }
    public int y()      { return y; }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Point other)) return false;
        return x == other.x && y == other.y;
    }

    @Override
    public int hashCode() { return Objects.hash(x, y); }

    @Override
    public String toString() { return "Point[x=" + x + ", y=" + y + "]"; }
}
```

A record reduces this to a single line and automatically generates everything above:

```java
// ✅ Java 16+ record — one line, complete implementation
public record Point(int x, int y) {}

// Usage:
Point p = new Point(3, 4);
System.out.println(p.x());       // 3  — auto-generated accessor (NOT getX(), just x())
System.out.println(p.y());       // 4
System.out.println(p);           // Point[x=3, y=4]  — auto-generated toString()
Point q = new Point(3, 4);
System.out.println(p.equals(q)); // true — auto-generated structural equality
```

A record is a special kind of class. It is **implicitly final** (cannot be extended), its components are **private final fields** (immutable), and the auto-generated constructor, accessors, `equals()`, `hashCode()`, and `toString()` all use the components.

#### Customising Records

Records are not just auto-generated boilerplate — they support extensive customisation.

```java
public record Range(int min, int max) {

    // Compact constructor — validates and transforms before fields are assigned.
    // You don't re-declare parameters; they are the record components.
    // Assignment to the fields happens automatically AFTER the body.
    Range {
        if (min > max) {
            throw new IllegalArgumentException(
                "min (" + min + ") must not exceed max (" + max + ")"
            );
        }
        // You can also normalise values here:
        // min = Math.max(0, min);  // Clamp negative min to 0
    }

    // Records can have additional instance methods
    public int length() {
        return max - min;
    }

    public boolean contains(int value) {
        return value >= min && value <= max;
    }

    public boolean overlaps(Range other) {
        return this.min <= other.max && other.min <= this.max;
    }

    // Records can have static methods and fields
    public static final Range EMPTY = new Range(0, 0);

    public static Range of(int min, int max) {
        return new Range(min, max);
    }

    // You can override the auto-generated toString() if you prefer a custom format
    @Override
    public String toString() {
        return "[" + min + ".." + max + "]";
    }
}

// Records can implement interfaces
public record Money(BigDecimal amount, Currency currency) implements Comparable<Money> {

    // Validation in compact constructor
    Money {
        Objects.requireNonNull(amount, "amount must not be null");
        Objects.requireNonNull(currency, "currency must not be null");
        if (amount.compareTo(BigDecimal.ZERO) < 0) {
            throw new IllegalArgumentException("amount must not be negative");
        }
        // Normalise scale to 2 decimal places
        amount = amount.setScale(2, RoundingMode.HALF_UP);
    }

    @Override
    public int compareTo(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException("Cannot compare different currencies");
        }
        return this.amount.compareTo(other.amount);
    }
}

// Generic records work just like generic classes
public record Pair<A, B>(A first, B second) {
    public Pair<B, A> swap() {
        return new Pair<>(second, first);
    }
}

Pair<String, Integer> pair = new Pair<>("hello", 42);
System.out.println(pair.first());           // "hello"
System.out.println(pair.swap().first());    // 42
```

> **When to use records:** Records are ideal for DTOs (Data Transfer Objects), response/request models in REST APIs, value objects in DDD, event objects, projection objects from database queries, and any class whose sole purpose is to carry immutable data. They work beautifully with Jackson (with the `jackson-module-parameter-names` or `@JsonCreator` annotation).

> **When NOT to use records:** Records are not suitable for entities that need to be mutable (JPA `@Entity` classes), classes with complex inheritance hierarchies, or classes whose `equals()` needs to be based on identity rather than structural equality.

### 20.2.5 Pattern Matching for `instanceof` (Preview Java 14–15, Stable Java 16)

Before this feature, checking whether an object was an instance of a type and then using it as that type required two steps: the `instanceof` test and an explicit cast. This pattern was so common it had its own name — "check-then-cast" — and it was a frequent source of ClassCastException bugs when the cast didn't match the test.

```java
// ❌ Classic pattern: check then cast — verbose and error-prone
Object obj = getShape();
if (obj instanceof Circle) {
    Circle c = (Circle) obj;  // The cast is redundant — we JUST checked it's a Circle
    double area = Math.PI * c.radius() * c.radius();
}
// Forgetting the cast means you can't call Circle-specific methods
// Mismatching the check and cast (Circle vs Square) is a runtime ClassCastException

// ✅ Java 16+ pattern matching: binding variable in the instanceof test
if (obj instanceof Circle c) {
    // 'c' is already typed as Circle — no cast needed, no risk of mismatch
    double area = Math.PI * c.radius() * c.radius();
}
```

The binding variable `c` is only in scope where the compiler can prove the test succeeded — inside the `if` block. The compiler also uses this for flow-sensitive type narrowing, which allows elegant negative patterns:

```java
// Flow-sensitive scoping: the binding variable is in scope wherever the test is guaranteed true
if (!(obj instanceof String s)) {
    // s is NOT in scope here — the test failed (or wasn't tested)
    return;
}
// s IS in scope here — we returned early if it wasn't a String
System.out.println(s.toUpperCase());  // Safe — compiler knows s is a String here

// Combining with logical operators
if (obj instanceof String s && s.length() > 5) {
    // s is in scope here — both conditions must be true to reach this block
    System.out.println(s.substring(0, 5));
}

// This does NOT compile — s is not definitely in scope when || is used
// because the test might not have been reached
// if (obj instanceof String s || s.length() > 5) { ... }  // ❌ compile error
```

Pattern matching for `instanceof` becomes even more powerful when combined with switch in Java 21 (covered in section 20.3). For now, the key benefit is eliminating explicit casts and making polymorphic code cleaner:

```java
// A shape processing method without pattern matching
double computeArea(Object shape) {
    if (shape instanceof Circle) {
        Circle c = (Circle) shape;
        return Math.PI * c.radius() * c.radius();
    } else if (shape instanceof Rectangle) {
        Rectangle r = (Rectangle) shape;
        return r.width() * r.height();
    } else if (shape instanceof Triangle) {
        Triangle t = (Triangle) shape;
        return 0.5 * t.base() * t.height();
    }
    throw new IllegalArgumentException("Unknown shape: " + shape);
}

// With Java 16 pattern matching: cleaner, no casts
double computeArea(Object shape) {
    if (shape instanceof Circle c)       return Math.PI * c.radius() * c.radius();
    if (shape instanceof Rectangle r)    return r.width() * r.height();
    if (shape instanceof Triangle t)     return 0.5 * t.base() * t.height();
    throw new IllegalArgumentException("Unknown shape: " + shape);
}
```

### 20.2.6 Helpful NullPointerExceptions (Java 14)

Before Java 14, a NullPointerException in a chained method call left you with a nearly useless message:

```
Exception in thread "main" java.lang.NullPointerException
    at com.example.OrderProcessor.process(OrderProcessor.java:42)
```

Line 42 contained `order.getCustomer().getAddress().getCity()`. Which one was null? You had no idea without a debugger. Java 14 made the JVM generate a precise message:

```
Cannot invoke "Address.getCity()" because the return value of "Customer.getAddress()" is null
```

This feature is enabled by default from Java 14 onwards. It works for field reads, method calls, array accesses, and unboxing — any place a NullPointerException can occur. This single feature has probably saved millions of developer-hours of debugging.

```java
// With helpful NPE messages (Java 14+), each of these tells you EXACTLY what was null:
user.getProfile().getAvatar().getUrl();
// Message: "Cannot invoke 'Avatar.getUrl()' because 'Profile.getAvatar()' is null"

int[] arr = null;
int x = arr[0];
// Message: "Cannot load from int array because 'arr' is null"

Integer boxed = null;
int unboxed = boxed;
// Message: "Cannot unbox 'boxed' because it is null"
```

## ☕ Java 15 — Sealed Classes Preview & Text Blocks Stable

### 20.2.7 Sealed Classes (Preview Java 15–16, Stable Java 17)

Sealed classes address a fundamental limitation: Java had no way to restrict which classes could extend a given class or implement a given interface. You could `final` a class to prevent *any* extension, or leave it open for *unlimited* extension — there was no middle ground. Sealed classes are that middle ground.

A **sealed class** explicitly lists which classes are permitted to extend it using the `permits` clause. This creates a closed type hierarchy — useful whenever a type has a fixed, known set of subtypes that represents a complete enumeration of possibilities.

```java
// Sealed class: exactly these three classes (and no others) can extend Shape
public sealed class Shape permits Circle, Rectangle, Triangle {
    public abstract double area();
}

// Each permitted subclass must be either:
// 1. final (no further extension)
// 2. sealed (extends the hierarchy with its own restrictions)
// 3. non-sealed (opens up extension again — explicitly opts out of sealing)

public final class Circle extends Shape {
    private final double radius;
    public Circle(double radius) { this.radius = radius; }
    @Override public double area() { return Math.PI * radius * radius; }
}

public final class Rectangle extends Shape {
    private final double width, height;
    public Rectangle(double width, double height) {
        this.width = width; this.height = height;
    }
    @Override public double area() { return width * height; }
}

public non-sealed class Triangle extends Shape {
    // non-sealed: anyone can extend Triangle further
    // This is the "escape hatch" for cases where a subtype needs to be extensible
    private final double base, height;
    public Triangle(double base, double height) {
        this.base = base; this.height = height;
    }
    @Override public double area() { return 0.5 * base * height; }
}

// Sealed interfaces work identically
public sealed interface Result<T> permits Success, Failure {
    record Success<T>(T value) implements Result<T> {}
    record Failure<T>(String error, Exception cause) implements Result<T> {}
}
```

**The key benefit of sealed classes** is that the compiler knows the complete, exhaustive set of subtypes. This enables **exhaustiveness checking** in switch expressions — the same way switching on all enum constants lets you omit `default`, switching on a sealed hierarchy lets you omit `default` when all permitted subtypes are covered. This is what makes pattern matching in switch (Java 21) so powerful.

```java
// Because Shape is sealed, the compiler can verify this switch is exhaustive
double area = switch (shape) {
    case Circle c      -> Math.PI * c.radius() * c.radius();
    case Rectangle r   -> r.width() * r.height();
    case Triangle t    -> 0.5 * t.base() * t.height();
    // No default needed — compiler knows these are ALL possible Shapes
};
```

> **When to use sealed classes:** They are ideal for modelling sum types (also called discriminated unions or tagged unions) — things like `Result<T>` (Success or Failure), `Shape` (Circle, Rectangle, Triangle), `PaymentStatus` (Pending, Approved, Declined, Refunded), or AST nodes in a compiler. Anywhere you have a type with a fixed, known set of meaningful variants, sealed classes give you compile-time safety that no amount of careful documentation can match.

## ☕ Java 16 — Records Stable and `Stream.toList()`

### 20.2.8 `Stream.toList()` (Java 16)

A small but frequently appreciated addition: `Stream.toList()` collects a stream into an unmodifiable `List<T>` with far less ceremony than `Collectors.toUnmodifiableList()`.

```java
// Before Java 16: verbose terminal operation
List<String> names = users.stream()
    .map(User::getName)
    .filter(Predicate.not(String::isBlank))
    .collect(Collectors.toUnmodifiableList());  // 40 characters of boilerplate

// Java 16+: concise and expressive
List<String> names = users.stream()
    .map(User::getName)
    .filter(Predicate.not(String::isBlank))
    .toList();  // Clean, obvious, immutable

// Important: the returned list is unmodifiable (throws UnsupportedOperationException on mutation)
// It does NOT preserve Comparator ordering unless you explicitly sorted the stream.
// It DOES allow null elements (unlike List.of()).
```

The `toList()` method is in the `Stream` interface directly — not in `Collectors`. It returns an **unmodifiable list** that preserves encounter order. This should be your default choice for collecting to a list when you don't need mutability.

## 💡 Phase 20.2 — Key Takeaways

The five releases from Java 12 to 16 form a coherent story about making Java more expressive and safe. Switch expressions replaced one of the language's most bug-prone constructs with a clean, exhaustive, expression-based form. Text blocks eliminated the string escaping nightmare that made JSON/SQL in Java code so painful. Records collapsed the definition of immutable data carriers from 50+ lines to 1. Pattern matching for `instanceof` removed the redundancy and risk of check-then-cast. Sealed classes gave type hierarchies a first-class formal contract. Each feature reinforces the others — records pair naturally with sealed classes, pattern matching works beautifully with sealed hierarchies, and switch expressions tie it all together. Java 17 LTS shipped all of them stable, which is why many people experience this as "Java finally got modern" when they upgrade from Java 8.
