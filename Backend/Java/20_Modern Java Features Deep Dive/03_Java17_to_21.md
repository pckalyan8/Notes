# 20.3 — Java 17 to 21 Features (LTS Milestones)

> **Versions covered:** Java 17 LTS (2021) · Java 18 (2022) · Java 19 (2022) · Java 20 (2023) · Java 21 LTS (2023)
>
> This era is defined by two transformative themes: the **completion** of the modern language features from Phase 20.2 (sealed classes, pattern matching in switch) and the **arrival** of Project Loom — Virtual Threads and Structured Concurrency — which fundamentally changes how Java handles concurrent I/O-bound workloads. Java 17 and Java 21 are both LTS releases, making this the most important upgrade path for most production systems today.

---

## ☕ Java 17 LTS — The Modern Java Baseline

Java 17 is considered the new Java 8 — the stable, widely-deployed LTS that teams upgrade to before jumping to the next LTS. It ships all features from Java 12–16 as stable, plus a few new ones.

### 20.3.1 Sealed Classes — Stable (Java 17)

Sealed classes (introduced as preview in Java 15–16) became stable in Java 17. The full syntax and semantics are documented in section 20.2.7. What's new to note here is that Java 17 also began the preview of **pattern matching in switch**, which directly leverages sealed class hierarchies for exhaustive matching. See section 20.3.3 for the full treatment.

---

### 20.3.2 Pattern Matching for `instanceof` — Stable (Java 16, first LTS: Java 17)

Also documented in section 20.2.5, this was stable since Java 16 but Java 17 is the first LTS where you can rely on it. It is worth reinforcing the most powerful pattern: using pattern matching to implement polymorphic behaviour without inheritance.

```java
// Polymorphic dispatch without requiring all types to implement a common interface
// Useful when processing types you don't own (third-party classes, Java library types)
String describe(Object obj) {
    if (obj instanceof Integer i)  return "Integer: " + i;
    if (obj instanceof Long l)     return "Long: " + l;
    if (obj instanceof Double d)   return "Double: %.2f".formatted(d);
    if (obj instanceof String s)   return "String of length " + s.length();
    if (obj instanceof int[] arr)  return "int array of length " + arr.length;
    return "Unknown: " + obj.getClass().getSimpleName();
}
```

### 20.3.3 Foreign Function & Memory API — Incubator (Java 17), Preview (Java 21), Stable (Java 22)

The **Foreign Function & Memory API** (Project Panama) addresses Java's historical weakness: calling native C/C++ libraries and managing off-heap memory. Before this API, the only option was JNI (Java Native Interface), which was notoriously complex, error-prone, and required writing C header files.

This API is incubating in Java 17 and covered in detail in section 20.4 (where it becomes stable in Java 22). We'll introduce the concept here so the progression makes sense.

The API revolves around two ideas. **MemorySegment** is a safe, bounded view of memory — either on-heap or off-heap — that prevents out-of-bounds access (something raw `Unsafe` couldn't guarantee). **Linker** connects Java method handles to native functions by name, without writing any C code.

## ☕ Java 18 — Smaller but Meaningful

### 20.3.4 UTF-8 as Default Charset (Java 18)

Before Java 18, the default charset — used by `new FileReader("file.txt")`, `new String(bytes)`, `System.out`, etc. — was platform-dependent. On Windows it was typically Windows-1252. On Linux it was typically UTF-8. This caused silent encoding bugs when code was developed on Mac/Linux and deployed on Windows, or when files were written on one platform and read on another.

Java 18 changed the default to **UTF-8 universally**, regardless of OS or locale. This eliminates an entire class of subtle, hard-to-diagnose encoding bugs.

```java
// Before Java 18: this could silently corrupt text containing non-ASCII characters
// if the platform default wasn't UTF-8
FileWriter writer = new FileWriter("output.txt");  // Uses platform default charset
writer.write("Ünïcödé täst");  // Might be garbled on Windows

// After Java 18: UTF-8 is always the default
FileWriter writer = new FileWriter("output.txt");  // Now always UTF-8
writer.write("Ünïcödé täst");  // Correct on all platforms

// If you need the old platform-dependent behaviour, explicitly specify the charset:
FileWriter writer = new FileWriter("output.txt", Charset.defaultCharset());

// Best practice remains: always specify charset explicitly for clarity
FileWriter writer = new FileWriter("output.txt", StandardCharsets.UTF_8);
```

### 20.3.5 `@snippet` Tag in Javadoc (Java 18)

The old `{@code ...}` inline Javadoc tag was designed for short inline code. For multi-line code samples, it was awkward — it didn't support syntax highlighting, couldn't reference actual source files, and had no validation. Java 18 introduces the `@snippet` block tag as a proper replacement.

```java
/**
 * Creates a new customer with the given details.
 *
 * <p>Example usage:
 *
 * {@snippet lang="java" :
 * CustomerService service = new CustomerService(repository);
 * Customer customer = service.createCustomer(
 *     "Alice",
 *     "alice@example.com",
 *     CustomerTier.GOLD
 * );
 * System.out.println(customer.id()); // prints the generated UUID
 * }
 *
 * <p>You can also reference a snippet from an external file:
 * {@snippet file="CreateCustomerExample.java" region="create-customer"}
 */
public Customer createCustomer(String name, String email, CustomerTier tier) { ... }
```

The `file=` attribute references a real source file in your `snippet-files/` directory, which means the snippet is **validated by the compiler** — if your example code doesn't compile, `javadoc` fails. This ends the problem of stale documentation examples.

### 20.3.6 Simple Web Server (Java 18)

Java 18 ships a minimal, zero-configuration HTTP file server accessible from the command line. It serves static files from a directory and is designed for development, prototyping, and testing — not production use.

```bash
# Serve the current directory on port 8000 (default)
$ jwebserver

# Serve a specific directory on a specific port
$ jwebserver -d /path/to/website -p 3000

# Bind to a specific address
$ jwebserver -b 0.0.0.0 -p 8080

# Output:
# Serving /path/to/website and subdirectories on 0.0.0.0 (all interfaces) port 8080
# URL: http://0.0.0.0:8080/
```

The Simple Web Server can also be launched programmatically — useful for integration tests that serve static files:

```java
import com.sun.net.httpserver.*;
import java.net.InetSocketAddress;
import java.nio.file.Path;

// Start a file server on port 8080 serving files from /tmp/test-assets
var server = SimpleFileServer.createFileServer(
    new InetSocketAddress(8080),
    Path.of("/tmp/test-assets"),
    SimpleFileServer.OutputLevel.INFO
);
server.start();
// ...
server.stop(0);
```

## ☕ Java 19–20 — Loom Previews and Concurrency Evolution

### 20.3.7 Virtual Threads (Preview Java 19–20, Stable Java 21)

Virtual Threads are the most significant Java feature since Java 8's lambdas and streams. To understand why, you need to understand the problem they solve.

**The problem with platform threads.** Traditional Java threads (called "platform threads" in Project Loom terminology) are thin wrappers around OS threads. Every OS thread requires a stack (typically 512 KB – 1 MB), and context-switching between threads is handled by the OS kernel, which involves a syscall overhead. This limits practical server applications to a few thousand concurrent threads. In a typical REST API that calls a database and maybe another service, a request occupies a thread for its entire duration — including all the time it's just waiting for the network or database to respond. This is "blocking" I/O, and it means most threads are doing nothing useful most of the time, yet still consuming OS resources.

Reactive frameworks (Project Reactor, RxJava) solved this by making everything non-blocking and callback-based — but at the cost of completely changing the programming model, making debugging very difficult, and requiring a different mental model for every developer on the team.

**Virtual Threads change the equation.** A Virtual Thread is scheduled by the JVM, not the OS. They are extremely cheap — millions can exist simultaneously. When a Virtual Thread blocks (e.g., waiting for a database query), the JVM "unmounts" it from the underlying OS thread (called the "carrier thread") and mounts a different Virtual Thread on that same OS thread. The OS thread never blocks waiting for I/O. This means you get the throughput benefits of reactive programming while keeping the simple, linear, synchronous programming model of classic Java.

```java
import java.util.concurrent.*;

// Creating a single virtual thread — just like platform thread but with ofVirtual()
Thread vThread = Thread.ofVirtual()
    .name("my-virtual-thread")
    .start(() -> {
        System.out.println("Running in virtual thread: " + Thread.currentThread());
        // This is a virtual thread — blocking here does NOT block an OS thread
        Thread.sleep(1000);
        System.out.println("Resumed after 1 second");
    });

vThread.join();  // Wait for it to complete

// The most common pattern: use a virtual thread per task executor
// This is the recommended way to use virtual threads with ExecutorService
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    // Submit 100,000 tasks — each gets its own virtual thread
    // This would have been impossible with platform threads (OOM or thread starvation)
    List<Future<String>> futures = IntStream.range(0, 100_000)
        .mapToObj(i -> executor.submit(() -> {
            // Simulate a blocking I/O call (database query, HTTP call)
            Thread.sleep(Duration.ofMillis(100));
            return "Result " + i;
        }))
        .collect(Collectors.toList());

    // Collect all results
    for (var future : futures) {
        System.out.println(future.get());
    }
}  // executor.close() blocks until all tasks complete, then shuts down
```

**Virtual threads and Spring Boot.** Spring Boot 3.2+ supports virtual threads natively. A single configuration property changes the entire application to use virtual threads for handling HTTP requests:

```yaml
# application.yml — Spring Boot 3.2+
spring:
  threads:
    virtual:
      enabled: true
```

With this configuration, Spring Boot uses `Executors.newVirtualThreadPerTaskExecutor()` as the web server's thread pool, allowing your application to handle many more concurrent requests without any code changes.

**Pinning — the key gotcha.** A Virtual Thread gets "pinned" to its carrier thread (unable to unmount) when it's inside a `synchronized` block or a `synchronized` method, or when calling native methods via JNI. During pinning, the carrier thread IS blocked — negating the advantage of virtual threads.

```java
// ❌ Pinning: synchronized blocks prevent unmounting
public synchronized void processPayment(Payment payment) {
    // The virtual thread is PINNED here — the carrier OS thread blocks
    // if this thread blocks inside (e.g., calling a database)
    database.save(payment);  // If this blocks, the carrier thread blocks too
}

// ✅ No pinning: use java.util.concurrent.locks.ReentrantLock instead
private final ReentrantLock lock = new ReentrantLock();

public void processPayment(Payment payment) {
    lock.lock();
    try {
        // Virtual thread can unmount here while waiting for the lock
        // Once acquired, it can also unmount during the database call
        database.save(payment);
    } finally {
        lock.unlock();
    }
}
```

You can detect pinning events with JFR (Java Flight Recorder):
```bash
java -XX:+UnlockDiagnosticVMOptions -XX:FlightRecorderOptions=stackdepth=128 \
     -Djdk.tracePinnedThreads=full MyApplication
```

> **Key Insight:** Virtual Threads don't make code faster per se — a single operation runs at the same speed. What they dramatically improve is **throughput under concurrent I/O-bound load**. If your bottleneck is CPU computation, virtual threads won't help. If your bottleneck is "threads waiting for database/HTTP responses", virtual threads can multiply your throughput several times over without changing your application's code.

### 20.3.8 Structured Concurrency (Incubator Java 19–21, Preview Java 21+)

Structured Concurrency introduces a programming model for concurrent code that mirrors structured programming (with its loops and if statements) — it gives concurrent tasks a clear, hierarchical relationship with defined lifetime boundaries.

The core problem it solves: when you split a task into multiple concurrent subtasks, managing errors and cancellations is complex and error-prone. If one subtask fails, how do you cancel the others? If the parent task is cancelled, how do you ensure all subtasks are cleaned up? Before Structured Concurrency, this required careful manual management of `CompletableFuture`, `CountDownLatch`, or similar primitives.

```java
import java.util.concurrent.*;

// The key class: StructuredTaskScope
// A scope creates a structured boundary — subtasks spawned inside it are "children"
// When the scope closes, ALL children are guaranteed to have completed (or been cancelled)

// Pattern 1: ShutdownOnFailure — cancel all siblings when any one fails
// Use this when you need ALL results to succeed
Response fetchUserData(long userId) throws InterruptedException, ExecutionException {
    try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {

        // Fork subtasks — these run concurrently, each in their own virtual thread
        Subtask<User>    userTask    = scope.fork(() -> userService.findById(userId));
        Subtask<List<Order>> ordersTask = scope.fork(() -> orderService.findByUserId(userId));
        Subtask<Address> addressTask = scope.fork(() -> addressService.findByUserId(userId));

        // Join: wait for all to complete (or for one to fail, or for the scope to be cancelled)
        scope.join()           // Wait for all forks to finish
             .throwIfFailed(); // If any fork threw an exception, re-throw it here

        // If we reach here, all tasks succeeded — access results safely
        return new Response(userTask.get(), ordersTask.get(), addressTask.get());

    } // scope.close() is called here — any still-running tasks are cancelled
}

// Pattern 2: ShutdownOnSuccess — cancel all siblings when any one succeeds
// Use this for "race to first result" scenarios (e.g., try multiple endpoints)
String fetchFromFastestSource(String query) throws InterruptedException, ExecutionException {
    try (var scope = new StructuredTaskScope.ShutdownOnSuccess<String>()) {

        scope.fork(() -> primaryService.search(query));
        scope.fork(() -> fallbackService.search(query));
        scope.fork(() -> cacheService.search(query));

        scope.join();  // Returns when the first one succeeds
        return scope.result();  // The first successful result

    } // Other tasks are automatically cancelled when scope closes
}
```

The beauty of Structured Concurrency is **observability**: because all concurrent work has a clear parent-child relationship, a thread dump shows the logical tree of concurrent work. This makes debugging concurrent systems dramatically easier — you can see which subtask belongs to which parent request.

## ☕ Java 21 LTS — The Landmark Release

Java 21 is the second LTS in this phase and ships the most feature-rich release since Java 8. It stabilises Virtual Threads, introduces Sequenced Collections, makes Record Patterns and Pattern Matching in switch both stable, and previews String Templates.

### 20.3.9 Pattern Matching in Switch (Stable Java 21)

Pattern matching in switch combines the expressiveness of switch expressions (section 20.2.1) with the power of pattern matching (section 20.2.5) and the exhaustiveness guarantee of sealed classes (section 20.2.7). The result is a feature that replaces entire categories of if-else chains and visitor patterns with a single, compiler-verified expression.

```java
// Basic type pattern in switch
String describe(Object obj) {
    return switch (obj) {
        case Integer i   -> "Integer: " + i;
        case Long l      -> "Long: " + l;
        case Double d    -> "Double: " + d;
        case String s    -> "String: '" + s + "'";
        case int[] arr   -> "int[] of length " + arr.length;
        case null        -> "null";       // Explicit null handling in switch
        default          -> "Other: " + obj.getClass().getSimpleName();
    };
}

// Guarded patterns (using 'when') — add conditions to pattern arms
// The 'when' clause is a boolean expression that further refines the pattern match
double processShape(Shape shape) {
    return switch (shape) {
        case Circle c when c.radius() > 100 -> {
            System.out.println("Large circle — applying special formula");
            yield Math.PI * c.radius() * c.radius();
        }
        case Circle c    -> Math.PI * c.radius() * c.radius();
        case Rectangle r -> r.width() * r.height();
        case Triangle t  -> 0.5 * t.base() * t.height();
        // No default needed — Shape is sealed, these three are exhaustive
    };
}

// Sealed hierarchy: exhaustive switch with no default
sealed interface Expr permits Num, Add, Mul {}
record Num(int value)       implements Expr {}
record Add(Expr left, Expr right) implements Expr {}
record Mul(Expr left, Expr right) implements Expr {}

int eval(Expr expr) {
    return switch (expr) {
        case Num(int n)          -> n;
        case Add(var l, var r)   -> eval(l) + eval(r);
        case Mul(var l, var r)   -> eval(l) * eval(r);
        // No default — compiler verifies Num, Add, Mul are all Expr subtypes
    };
}
```

### 20.3.10 Record Patterns (Stable Java 21)

Record Patterns extend pattern matching to deconstruct records directly in `instanceof` tests and `switch` expressions. Instead of matching a type and then manually calling accessors, you can match the type and destructure its components in a single pattern.

```java
// Without record patterns: match then manually access components
if (obj instanceof Point p) {
    int x = p.x();   // Manual accessor calls
    int y = p.y();
    System.out.printf("Point at (%d, %d)%n", x, y);
}

// With record patterns (Java 21): match and destructure in one step
if (obj instanceof Point(int x, int y)) {
    // x and y are directly in scope — no accessor calls needed
    System.out.printf("Point at (%d, %d)%n", x, y);
}

// Record patterns in switch — this is where they really shine
sealed interface Shape permits Circle, Rectangle, Triangle {}
record Circle(double radius) implements Shape {}
record Rectangle(double width, double height) implements Shape {}
record Triangle(double base, double height) implements Shape {}

double area(Shape shape) {
    return switch (shape) {
        case Circle(double r)              -> Math.PI * r * r;
        case Rectangle(double w, double h) -> w * h;
        case Triangle(double b, double h)  -> 0.5 * b * h;
    };
}

// Nested record patterns — destructure records that contain records
record Order(int id, Customer customer, List<Item> items) {}
record Customer(String name, Address address) {}
record Address(String city, String country) {}

String getOrderCity(Object obj) {
    return switch (obj) {
        // Nested destructuring: match Order containing Customer containing Address
        case Order(int id, Customer(String name, Address(String city, var country)), var items)
            -> "Order %d from %s is in %s, %s".formatted(id, name, city, country);
        default -> "Not an order";
    };
}
```

### 20.3.11 Sequenced Collections (Java 21)

Before Java 21, the Collections Framework had an embarrassing gap: there was no common way to access the **first** or **last** element of a collection with a defined encounter order. Each collection had its own API:

```java
// Before Java 21: inconsistent first/last access across collection types
List<String> list      = new ArrayList<>(List.of("a", "b", "c"));
String first_list      = list.get(0);              // List
String last_list       = list.get(list.size() - 1);

LinkedHashSet<String> set = new LinkedHashSet<>(Set.of("a", "b", "c"));
String first_set       = set.iterator().next();    // Set — verbose!
// Getting last from a LinkedHashSet before Java 21 was painful

Deque<String> deque    = new ArrayDeque<>(List.of("a", "b", "c"));
String first_deque     = deque.peekFirst();        // Deque — different method
String last_deque      = deque.peekLast();
```

Java 21 introduces three new interfaces that provide a unified API:

**`SequencedCollection<E>`** extends `Collection<E>` and adds `getFirst()`, `getLast()`, `addFirst()`, `addLast()`, `removeFirst()`, `removeLast()`, and `reversed()`. It is now implemented by `List`, `Deque`, and `LinkedHashSet`.

**`SequencedSet<E>`** extends both `SequencedCollection<E>` and `Set<E>`. It is implemented by `LinkedHashSet`, `SortedSet`, and their subclasses.

**`SequencedMap<K,V>`** extends `Map<K,V>` and adds `firstEntry()`, `lastEntry()`, `pollFirstEntry()`, `pollLastEntry()`, `putFirst()`, `putLast()`, and `reversed()`. It is implemented by `LinkedHashMap`, `SortedMap`, and `NavigableMap`.

```java
import java.util.*;

List<String> list = new ArrayList<>(List.of("alpha", "beta", "gamma"));

// Java 21 unified API — same methods work on List, Deque, LinkedHashSet
list.getFirst();              // "alpha"
list.getLast();               // "gamma"
list.addFirst("zero");        // ["zero", "alpha", "beta", "gamma"]
list.addLast("delta");        // ["zero", "alpha", "beta", "gamma", "delta"]
list.removeFirst();           // removes and returns "zero"
list.removeLast();            // removes and returns "delta"

// reversed() returns a reversed VIEW — not a copy
List<String> reversed = list.reversed();
System.out.println(reversed.getFirst());  // "gamma"
// Modifications to reversed are reflected in the original and vice versa

// SequencedMap — same consistency for maps with defined order
LinkedHashMap<String, Integer> scores = new LinkedHashMap<>();
scores.put("Alice", 95);
scores.put("Bob", 87);
scores.put("Carol", 92);

Map.Entry<String, Integer> first = scores.firstEntry();  // Alice=95
Map.Entry<String, Integer> last  = scores.lastEntry();   // Carol=92

// putFirst/putLast let you insert in specific positions
scores.putFirst("Zara", 99);   // Zara goes to the front

// reversed() on a map gives a view with reversed iteration order
scores.reversed().forEach((k, v) -> System.out.println(k + "=" + v));
// Output: Carol=92, Bob=87, Alice=95, Zara=99
```

### 20.3.12 String Templates (Preview Java 21)

String Templates bring string interpolation to Java — the ability to embed expressions directly inside string literals without calling `String.format()` or `+` concatenation. They are previewing in Java 21; coverage of their evolution into Java 22+ is in section 20.4.

> **Note:** String Templates were re-evaluated after their Java 21 and Java 22 preview, and in Java 23 the preview was withdrawn to be redesigned. The design documented here reflects the Java 21 preview API (`STR` template processor). The feature is being reconsidered for a future release. Learn the concept but do not use `STR` in production code yet.

```java
// The concept: embed expressions with \{...} inside template strings
// The STR template processor evaluates and interpolates them

String name = "Alice";
int age = 30;

// STR is a built-in template processor
String greeting = STR."Hello, \{name}! You are \{age} years old.";
// Result: "Hello, Alice! You are 30 years old."

// Expressions can be any Java expression
double price = 49.99;
int quantity = 3;
String receipt = STR."Total: $\{price * quantity}";
// Result: "Total: $149.97"

// Multi-line templates (combined with text blocks)
String json = STR."""
        {
            "name": "\{name}",
            "age": \{age},
            "active": \{age >= 18}
        }
        """;

// The FMT processor applies printf-style formatting to embedded expressions
import static java.util.FormatProcessor.FMT;
String formatted = FMT."Price: $%.2f\{price}";
// Result: "Price: $49.99"
```

The deeper design allows custom template processors — you can create a `SQL` processor that sanitises inputs to prevent injection, a `JSON` processor that properly escapes values, or an `HTML` processor that entity-encodes content. This is the feature's most powerful capability.

## 💡 Phase 20.3 — Key Takeaways

Java 17 to 21 represents Java's most ambitious five-year period since the generics era. Java 17 LTS stabilised the full modern language feature set from Phase 20.2. Java 18 fixed the long-standing UTF-8 encoding inconsistency. The path from Java 19 to 21 delivered Virtual Threads and Structured Concurrency, giving Java a first-class answer to reactive frameworks for I/O-bound workloads — without sacrificing the simplicity of sequential, blocking code. Java 21 LTS is the upgrade destination for any team still on Java 11 or Java 17; it ships a more expressive switch with patterns, first-class record destructuring, and a unified first/last API for collections. For most teams, upgrading to Java 21 is the highest-leverage platform investment available today.
